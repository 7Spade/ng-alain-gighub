## 專案願景

### 產品定位

建築工程管理系統是一個基於 **Git-like 分支模型**的協作管理平台，專為建築工程行業設計，支援組織間的承攬關係管理、任務執行追蹤、品質驗收與責任切割。

### 核心價值主張

1. **Git-like 分支模型**：主分支（擁有者）與組織分支（承攬商）權限分離，確保結構控制權
2. **承攬關係管理**：1:1 組織協作關係，明確權責劃分
3. **責任切割機制**：透過暫存區、品管、驗收流程，明確責任轉移點
4. **跨組織協作**：支援多組織協同作業，數據即時同步至主分支
5. **視覺化追蹤**：進度儀表板、待辦中心、活動記錄全面掌控

## 使用者痛點

### 1. 多組織協作管理問題

#### 業主（建設公司）面臨的挑戰

- **難以統一管理多家承攬商的工作進度**
  - 數據散落各處，缺乏統一視圖
  - 無法全局掌控專案狀況
  - 進度追蹤不準確

- **責任劃分不清**
  - 出現問題時難以追溯
  - 責任歸屬爭議頻繁
  - 缺乏明確的責任切割點

- **數據格式不一致**
  - 承攬商提交的數據格式各異
  - 審核成本高
  - 數據品質難以保證

### 2. 任務執行與追蹤問題

#### 任務管理面臨的挑戰

- **任務執行流程複雜**
  - 缺乏統一標準
  - 流程不清晰
  - 執行效率低

- **數據散落各處**
  - 難以即時掌握狀況
  - 資訊不透明
  - 協調成本高

- **進度追蹤不準確**
  - 無法預測完工時間
  - 資源分配困難
  - 任務優先順序不明確

### 3. 品質驗收與責任切割問題

#### 品質管理面臨的挑戰

- **驗收流程不明確**
  - 責任轉移點不清晰
  - 驗收標準不一致
  - 驗收記錄易遺失

- **品管記錄管理困難**
  - 照片管理混亂
  - 缺失追蹤不完整
  - 改善措施難以落實

- **責任歸屬爭議**
  - 缺乏明確的責任切割點
  - 責任轉移時間點不明確
  - 追溯困難

### 4. 問題追蹤與異常處理問題

#### 異常處理面臨的挑戰

- **施工異常問題難以追蹤**
  - 跨組織溝通不順
  - 問題狀態不明確
  - 處理進度無法掌握

- **問題散落各分支**
  - 擁有者無法全局掌控
  - 問題解決方案缺乏記錄
  - 類似問題重複發生

### 5. 協作溝通與待辦管理問題

#### 協作溝通面臨的挑戰

- **資訊散落各處**
  - 難以統一管理待辦事項
  - 通知機制不完善
  - 重要訊息容易遺漏

- **討論記錄不完整**
  - 決策過程難以追溯
  - 待辦事項狀態不明確
  - 優先順序混亂

### 6. 數據分析與視覺化問題

#### 數據分析面臨的挑戰

- **缺乏統一的數據視圖**
  - 報表生成耗時
  - 無法即時查看
  - 跨分支數據難以比對

- **歷史數據查詢困難**
  - 趨勢分析不準確
  - 審計追溯困難
  - 活動記錄不完整

## 商業動機

### 市場機會

- **建築工程行業數位化轉型**
  - 傳統管理方式效率低下
  - 數位化工具需求增長
  - 競爭優勢明顯

- **SaaS 模式優勢**
  - 降低企業 IT 成本
  - 快速部署和擴展
  - 持續更新和優化

### 商業模式

#### 方案 1：基礎版（Free）
- **價格**：免費
- **限制**：1 個組織、3 個藍圖、5 個成員、1 GB 儲存空間
- **適用對象**：小型工作室、個人使用

#### 方案 2：專業版（Pro）
- **價格**：NT$ 3,000/月（年繳 NT$ 30,000）
- **功能**：無限組織、無限藍圖、50 個成員、100 GB 儲存空間
- **適用對象**：中小型營造廠

#### 方案 3：企業版（Enterprise）
- **價格**：客製報價（約 NT$ 50,000/月起）
- **功能**：專業版所有功能 + 無限成員、無限儲存、專屬支援
- **適用對象**：大型建設公司、營造集團

## 技術動機

### 技術選型原因

#### 為什麼選擇 Git-like 分支模型？

- **權限分離清晰**：主分支擁有者完全控制任務結構，組織分支只能填寫承攬欄位
- **數據追溯性強**：所有變更通過 PR 機制審核，完整記錄變更歷史
- **協作流程標準化**：類似 Git 的工作流程，降低學習成本

#### 為什麼選擇 Angular 20 + ng-alain？

- **企業級框架**：ng-alain 提供完整的企業級功能（權限、布局、路由等）
- **現代化語法**：Angular 20 的 Standalone Components、Signals 等新特性
- **豐富的 UI 組件**：NG-ZORRO 提供完整的 Ant Design 組件庫

#### 為什麼選擇 Supabase？

- **全棧解決方案**：Auth、Database、Storage、Realtime 一站式服務
- **PostgreSQL 原生支持**：強大的 RLS 功能，確保數據安全
- **開發效率高**：自動生成 API、類型定義，減少開發工作量

## 問題定義

### 核心問題

**如何讓擁有者保持對專案結構的控制權，同時讓承攬商能夠獨立執行任務並提交數據，最終通過 PR 機制實現數據統一和審核流程？**

### 解決方案概述

1. **Git-like 分支模型**：主分支控制結構，組織分支執行任務
2. **權限分離架構**：從資料庫層（RLS）到應用層雙重驗證
3. **PR 機制**：承攬商提交 → 擁有者審核 → 合併更新
4. **暫存區機制**：48 小時可撤回，減少錯誤提交
5. **責任切割機制**：通過品管、驗收流程明確責任轉移點
